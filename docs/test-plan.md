# 测试计划

这个文件定义 MVP 阶段如何验证系统可靠性。首版要避免测试强依赖真实外部 API 和真实模型返回，因此必须优先使用 fixtures、stub 和 mock。

## 必跑检查

- 单元测试：指标计算模块、代理配置解析、数据标准化、报告 schema 组装
- 集成测试：使用模拟数据源验证“抓取 -> ETL -> 报告生成”链路
- 端到端冒烟测试：访问 Web 页面并触发一次分析，返回报告页面或 JSON 结果
- Lint：后续项目初始化后补充
- Build：Python 项目首轮以“可启动 + 测试通过”替代单独 build 流程

## 关键路径测试

1. 启动应用
2. 进入分析页面
3. 选择默认资产和周期
4. 触发一次分析
5. 使用 mock 数据或 fixture 数据完成情报聚合
6. 成功返回结构化报告
7. 报告中包含多头逻辑、空头逻辑、概率分布和失效条件

## 测试数据策略

- 默认提供一组本地 fixture 数据，覆盖行情、宏观和事件文本
- 集成测试中默认禁用真实外部网络请求
- 多智能体测试默认使用可预测的 stub 返回，避免测试不稳定

## 失败处理策略

- 如果必跑检查失败，先修复失败项，再继续开发新功能
- 修复 bug 时优先新增回归测试
- 如果失败来自外部 API 或代理问题，应保留 mock 路径保证本地测试可继续执行
